### 2026-04-26

### System synthesis

Claude / Gemini / Codex Awwwards portfolio system

### What improved

- Reframed the research from a Ballet Edmonton page pass into a portfolio-wide art-direction system.
- Adopted the skeleton / wardrobe model as the reusable project-page architecture.
- Adopted the Single Author model so all pages feel governed by one portfolio identity.
- Added energy categories to control project-specific variation without fragmenting navigation or interaction logic.
- Defined Ballet Edmonton as pilot validation only.

### Direction chosen

One authored portfolio system with controlled variation.

Same skeleton. Different wardrobe. One author.

### Rules learned from the synthesis

- The portfolio is judged at its weakest page, not its strongest page.
- Project pages share a fixed skeleton but choose different presentation modules.
- Project-specific art direction must not become project-specific navigation.
- Energy categories can change tempo, density, and motion intensity, but not the site's identity.
- Prepared screenshots are finished artwork; if they do not fit a slot, change the slot.
- Technical/system projects need precision plus at least one emotional or cinematic image break.
- Motion previews belong only where motion is central, and they need static poster fallbacks.
- Every animation must earn its place.

### Ballet Edmonton as pilot validation

Ballet Edmonton validates the system for a calm, editorial, mixed-media project. It should not define the default style for every project.

What the pilot taught:

- Calm/editorial pacing works for cultural work.
- Screenshot scale must respect source quality.
- Website sections can quickly become gallery-like if the layout becomes too grid-driven.
- Mixed-media projects need one throughline, not equal treatment for every medium.

### What should not become default

- Do not make every project as quiet as Ballet Edmonton.
- Do not use Ballet's screenshot treatment as the universal web/digital layout.
- Do not let the pilot's calm motion tone define campaign, motion, or large-format work.
- Do not turn screenshot integrity into an excuse for showing every screenshot at equal weight.

### How future project pages should use the system

1. Identify the project type.
2. Choose the energy category.
3. Select the hero archetype.
4. Build the page skeleton.
5. Choose project-specific wardrobe modules.
6. Sequence assets by story role.
7. Define one possible memorable interaction or visual moment.
8. Protect mobile, performance, and accessibility.
9. Record learnings after the page ships.

### Final direction

Create a portfolio-wide motion system for project browsing, not page-specific choreography.

Motion should:

- clarify hierarchy
- improve orientation
- support storytelling
- express project energy through tempo
- preserve the Single Author model
- protect performance and accessibility

### Rules learned

- Static preview is the baseline for project index and next-project modules.
- Motion preview is the exception, reserved for motion-led projects.
- Hover must equal keyboard focus.
- Mobile must not depend on hover.
- Energy categories may change timing and intensity, but not interaction architecture.
- Prepared screenshots should receive wrapper/plate reveals only, not internal animation.
- Reduced-motion is required before adding more project ecosystem motion.
- No page should feel like every section is animated.

### Ballet Edmonton pilot motion notes

- Energy category: Calm / Editorial with Mixed Media support.
- Keep existing hero handoff, panel wipe, static screenshot plates, and restrained section reveal.
- Do not add Remotion for the website showcase.
- Do not crop, zoom, mask, or animate prepared website screenshots.
- Any future Ballet motion should feel like controlled atmosphere and hierarchy, not literal dance movement.

### What should not become default

- Do not apply Ballet Edmonton's calm tempo to high-energy campaign pages.
- Do not apply universal parallax to every image without project-specific justification.
- Do not add project index motion previews globally.
- Do not make screenshot sections interactive demos by default.

### Next implementation priority

Before adding new motion, implement the foundation:

1. Add reduced-motion support.
2. Add focus-visible parity for existing hover states.
3. Add project data fields for `energyCategory`, `projectType`, `posterImage`, and optional `previewVideo`.
4. Refine work index preview architecture.
5. Strengthen next-project module as a consistent transition pattern.

- Route-level foundations matter creatively because broken skip links, missing 404s, or generic metadata weaken perceived polish.
- Performance should be treated as an art-direction constraint, not only an engineering cleanup.
- Motion should not expand until reduced-motion and mobile behavior are complete.
- Project pages should be prioritized by submission risk, not by how recently they were edited.

- Page The Cleaners needs the most structural uplift because signage/environmental work requires scale, context, and final payoff.
- Edify needs a performance-safe motion presentation system before it can carry a motion/experimental category.

- Ballet Edmonton needs only targeted polish for now; do not keep iterating on it while broader blockers remain.

# Merged Synthesis Integration

Date added: 2026-04-27

## What was learned from Claude

- The portfolio needed a strict structural system, not taste-driven one-offs.
- The skeleton / wardrobe model is the correct architecture for keeping pages consistent without making them identical.
- Project-type modules need to be documented explicitly so future work does not restart the same design questions.
- Accessibility, usability, and next-project continuity must be treated as part of art direction, not an afterthought.

## What was learned from Gemini

- The repo needed more material specificity around lighting, texture, screenshot treatment, and what reads as premium versus cheap.
- Mockup critique had to become concrete:
  real photography, believable surface contact, and consistent light outperform template polish.
- Performance and media delivery standards needed clearer practical targets.
- Trend labels and reference scanning are only useful when translated into durable visual rules.

## What was accepted

- skeleton / wardrobe model
- single-author portfolio system
- fixed five-act project-page structure
- project-type wardrobe modules
- prepared screenshot integrity
- real-photography-first hierarchy
- one mockup philosophy per project page
- every-animation-earns-its-place rule
- weakest-page submission mindset

## What was rejected

- decorative mouse trails
- ASCII reveal gimmicks
- scroll-jacking as a feature
- floating laptop/phone composites
- fake generated UI
- generic PSD mockups
- fake luxury styling
- trend-label thinking
- using Ballet Edmonton's tone as a universal page answer

## What was deferred

- Remotion or pre-rendered motion except where a project genuinely requires frame-perfect motion
- advanced cursor systems
- additional broad audit passes before more production work
- non-essential interaction novelty

## How this changes Ballet Edmonton

- Ballet Edmonton remains the pilot validation page, not the house style.
- Its prepared website screenshots remain protected as finished artwork.
- Its calm editorial rhythm remains appropriate for that project only.
- Further Ballet work should focus on production-grade refinement, not more abstract system discovery.

## How this changes all future project pages

- Future pages start from the documented skeleton first.
- Wardrobe decisions are selected by project type instead of improvised.
- Hero, sequencing, mockup, screenshot, and motion decisions are now judged against an explicit production standard.
- Weak assets should be removed sooner.
- The next phase should prioritize visual production quality over more portfolio-wide audits.

## New priority

Visual production now has priority over more broad auditing. The repo has enough system guidance to move into project-page upgrades, asset replacement, mockup production, and route-by-route visual refinement.
